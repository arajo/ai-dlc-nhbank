# 테이블오더 서비스 - 요구사항 정의서

## Intent Analysis Summary

| 항목 | 내용 |
|------|------|
| **사용자 요청** | 테이블오더 서비스 구축 (고객 주문 + 관리자 모니터링) |
| **요청 유형** | New Project (신규 프로젝트) |
| **범위** | Multiple Components (프론트엔드 + 백엔드 + 데이터베이스) |
| **복잡도** | Moderate (다수 기능, 실시간 통신, 멀티 인터페이스) |
| **프로젝트 유형** | Greenfield |

---

## 1. 프로젝트 개요

### 1.1 서비스 비전
단일 매장 환경에서 고객에게 편리한 디지털 주문 경험을, 매장 운영자에게 효율적인 실시간 주문 관리 환경을 제공하는 테이블오더 플랫폼.

### 1.2 운영 환경
- **대상**: 단일 매장 (1개 매장)
- **규모**: 소규모 (1~10 테이블)
- **동시 접속**: 10명 이하
- **관리자**: 단일 관리자 계정 (매장당 1개)

---

## 2. 기술 스택

| 영역 | 기술 | 비고 |
|------|------|------|
| **백엔드** | Python (FastAPI) | 비동기 지원, SSE 네이티브 |
| **프론트엔드** | React | 고객용 + 관리자용 SPA |
| **데이터베이스** | SQLite | 경량, 파일 기반, 로컬 개발 최적 |
| **인증** | JWT | 16시간 세션, bcrypt 해싱 |
| **실시간 통신** | Server-Sent Events (SSE) | 주문 실시간 업데이트 |
| **이미지 관리** | 외부 URL 직접 입력 | 별도 이미지 호스팅 사용 |
| **타겟 디바이스** | 반응형 웹 (모든 화면 크기) | 태블릿 + 스마트폰 + PC |
| **배포** | 로컬 개발 우선 (배포 환경 추후 결정) | — |
| **PBT 프레임워크** | Hypothesis (Python), fast-check (JS/TS) | PBT 확장 활성화 |

## 3. 기능 요구사항 (Functional Requirements)

### 3.1 고객용 기능 (Customer Interface)

#### FR-C01: 테이블 태블릿 자동 로그인 및 세션 관리
- 관리자가 1회 초기 설정 (매장ID, 테이블번호, 비밀번호)
- 로그인 정보 로컬 저장
- 이후 자동 로그인 (저장된 정보 사용)
- 16시간 세션 유지

#### FR-C02: 메뉴 조회 및 탐색
- 메뉴 화면이 기본(홈) 화면
- 카테고리별 메뉴 분류 및 표시
- 메뉴 상세 정보: 메뉴명, 가격, 설명, 이미지(외부 URL)
- 카드 형태 레이아웃
- 카테고리 간 빠른 이동
- 터치 친화적 버튼 (최소 44x44px)

#### FR-C03: 장바구니 관리
- 메뉴 추가/삭제
- 수량 조절 (증가/감소)
- 총 금액 실시간 계산
- 장바구니 비우기
- 클라이언트 로컬 저장 (새로고침 시에도 유지)
- 서버 전송은 주문 확정 시에만

#### FR-C04: 주문 생성
- 주문 내역 최종 확인
- 주문 확정 버튼
- 주문 성공 시: 주문번호 표시 → 장바구니 비우기 → 5초 후 메뉴 화면 자동 리다이렉트
- 주문 실패 시: 에러 메시지 표시 + 장바구니 유지
- 주문 정보: 매장ID, 테이블ID, 메뉴목록(메뉴명/수량/단가), 총금액, 세션ID

#### FR-C05: 주문 내역 조회
- 주문 시간순 정렬
- 주문별 상세: 주문번호, 시각, 메뉴/수량, 금액, 상태(대기중/준비중/완료)
- 현재 테이블 세션 주문만 표시 (이전 세션 제외)
- 매장 이용 완료된 주문 제외

### 3.2 관리자용 기능 (Admin Interface)

#### FR-A01: 매장 인증
- 매장 식별자 + 사용자명 + 비밀번호 로그인
- 16시간 세션 유지 (JWT)
- 브라우저 새로고침 시 세션 유지
- 16시간 후 자동 로그아웃
- 비밀번호 bcrypt 해싱
- 로그인 시도 제한

#### FR-A02: 실시간 주문 모니터링
- SSE 기반 실시간 주문 업데이트
- 그리드/대시보드 레이아웃: 테이블별 카드 형태
- 각 테이블 카드: 총 주문액, 최신 주문 n개 미리보기
- 주문 카드 클릭 시 전체 메뉴 상세 보기
- 주문 상태 변경 (대기중→준비중→완료)
- 신규 주문 시각적 강조 (색상/애니메이션)
- 2초 이내 주문 표시
- 테이블별 필터링

#### FR-A03: 테이블 관리
- **초기 설정**: 테이블 번호 및 비밀번호 설정, 16시간 세션 생성
- **주문 삭제**: 확인 팝업 → 즉시 삭제 → 총 주문액 재계산 → 피드백
- **테이블 세션 처리**: 이용 완료 시 주문 이력 이동 → 현재 주문 리셋 → 새 고객 대비
- **과거 내역 조회**: 테이블별 과거 주문 목록 (시간 역순), 날짜 필터링

#### FR-A04: 메뉴 관리
- 메뉴 CRUD (조회/등록/수정/삭제)
- 메뉴 정보: 메뉴명, 가격, 설명, 카테고리, 이미지 URL
- 메뉴 노출 순서 조정
- 필수 필드 검증
- 가격 범위 검증

## 4. 비기능 요구사항 (Non-Functional Requirements)

### NFR-01: 성능
- SSE 기반 주문 알림: 2초 이내 전달
- 페이지 로딩: 3초 이내

### NFR-02: 사용성
- 반응형 웹 디자인 (모든 화면 크기)
- 터치 친화적 UI (최소 44x44px 터치 영역)
- 직관적 시각적 계층 구조

### NFR-03: 데이터 무결성
- 주문 생성 시 원자적 처리
- 세션 종료 시 데이터 일관성 보장
- 장바구니 로컬 영속성

### NFR-04: 가용성
- 단일 서버 운영 (소규모 단일 매장)
- SQLite 파일 기반 자동 백업 (정기적)
- 서비스 재시작 시 데이터 보존

---

## 5. 회복성 요구사항 (Resiliency Requirements)

### RES-01: 복구 목표
- **RTO/RPO**: N/A (단일 리전/단일 서버, 멀티존 불필요)
- **DR 전략**: 로컬 백업 기반 복원 (SQLite 파일 백업)

### RES-02: 변경 관리
- AI-DLC가 제안하는 경량 변경 관리 프로세스 채택
- 변경 기록 + 승인 + 롤백 노트 구조

### RES-03: CI/CD 및 배포
- AI-DLC가 제안하는 CI/CD 파이프라인 사용
- **배포 전략**: Canary (점진적 트래픽 이동 + 자동 롤백)
- **롤백 메커니즘**: Blue/green 스왑 (이전 환경 복귀)

### RES-04: 인시던트 대응
- AI-DLC가 제안하는 경량 인시던트 대응 + COE 프로세스 채택

---

## 6. 테스트 요구사항 (Testing Requirements)

### TEST-01: Property-Based Testing
- **적용 범위**: 전체 (Full enforcement)
- **Python 프레임워크**: Hypothesis
- **JavaScript 프레임워크**: fast-check
- **대상 영역**: 비즈니스 로직, 데이터 변환, 직렬화, 상태 관리(장바구니, 주문)

---

## 7. 제외 범위 (Out of Scope)

다음 기능은 구현하지 않습니다:
- 결제 처리 (카드, 현금, PG사 연동)
- 영수증 발행/환불
- 포인트/쿠폰 시스템
- OAuth/SNS 로그인, 2FA
- 이미지 업로드/리사이징
- 푸시/SMS/이메일 알림
- 주방 시스템 연동
- 데이터 분석/매출 리포트
- 재고/직원/예약/리뷰/다국어
- 외부 플랫폼 연동 (배달, POS, 소셜)
